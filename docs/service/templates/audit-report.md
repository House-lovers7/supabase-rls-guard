# 監査報告書テンプレート

RLSスポット監査（[docs/service/README.md](../README.md)）の納品物テンプレート。
記入例は [合成サンプル](../samples/synthetic-audit-report.md) を参照。

## 1. 概要

- 対象: （リポジトリ / ディレクトリ、ファイル数）
- 実施期間: YYYY-MM-DD 〜 YYYY-MM-DD
- 使用ツール: supabase-rls-guard vX.Y.Z（パーサーバックエンド: libpg / regex）
- 実施者:

## 2. スキャン完全性

- スキャンしたファイル数:
- 解析警告数: ____ 件（parser fallback / 読み取り不能エントリ / 設定不備）
- 解析警告が1件以上ある場合、該当範囲は**監査未完了**として第4節に列挙する。

## 3. 検出結果と人手判定

ツール出力（CLIルールID・CLI重大度）と人手判定を分けて記載する。

| # | CLIルールID | 対象 | CLI重大度 | 人手判定 | 根拠 | 推奨対応 |
| - | ----------- | ---- | --------- | -------- | ---- | -------- |
| 1 | RLS0NN | schema.table | Critical | 真陽性 / 偽陽性 / 意図的公開 |  |  |

人手判定の区分:

- **真陽性**: 対応が必要な設定ミス
- **偽陽性**: ツールの静的近似による誤検出（根拠を必ず記載する）
- **意図的公開**: 仕様として公開しているもの（`publicTables` での明示を推奨）

## 4. 監査未完了の範囲

解析警告により静的解析が完全でない範囲（ファイル・ステートメント）と、補完の
ための推奨手段（ライブ DB での Supabase Security Advisor 実行等）を列挙する。
該当がない場合は「なし」と明記する。

## 5. 再確認結果（再確認実施時のみ記入）

- 再確認日: YYYY-MM-DD（初回納品から30日以内・1回）
- 対象: 初回報告書の指摘 #
- 再確認結果: 解消 / 未解消 / 新規の解析警告（詳細: ）

## 6. 免責

本報告書は静的解析と人手レビューの範囲での指摘であり、対象システムに問題が
存在しないことを示すものではありません。ライブデータベースの状態（ダッシュ
ボードから作成されたスキーマ等）は対象外です。
